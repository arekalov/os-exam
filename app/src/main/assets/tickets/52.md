## Feedback-планировщик
* Для каждого уровня приоритета есть своя очередь готовых процессов и свой квант времени. Очереди обслуживатюся по FIFO внутри одного приоритета.
* Если прцоесс активно занимает процессор и полностью выбирает квант, его приоритет понижают -> процесс спускает вниз по таблце приоритетов, но получает больший квант времени
* Если процесс много ждет IO и блокируется, при возврате из ожидания ему резко повышают приоритет, т.к. такие процессы должны быстро обработать полученные данные
* В Solaris для пользоват. класса расписана таблца, в котор. для кажд. приоритеа указаны: 
  * размер кванта
  * новый приоритет после исчерпания кванта
  * новый приоритет после блокировки/ожидания
  * поведение при долгом ожидании
* max wait - граничное время ожидания, после которого приоритет поднимали еще раз, сейчас почти не использ.

## Классы планирования
* **TimeSharing** - основной пользовательский класс, реализующ. описанный фидбэек-алгоритм. Все пользоват. процессы старутуют здесь с приоритет. 0-59
* **InterActive** - аналог. TS, но с повышен. приоритет. для активного GUI терминального процесса, чтобы активное приложение откликалось быстрее
* **Fixed**, **System**, **Realtime** - классы с фикс. приоритетами, не меняющ. динамич. Для кажд. приоитета задан свой квант, но никакого изменения приоритета по таблце внутри класса нет
* **Fair Share Scheduler** - справедливый планировщик, всем процессам дается числ. занчение и каждый процесс получает в среднем долю процессора, совпад. с долей его числа в сумме всех чисел процессов
* Отдельные приоритеты для Interrupt threads
* Учет **афинити** для NUMA
